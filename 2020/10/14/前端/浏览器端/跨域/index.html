跨域的方法

在浏览器中只要 协议 端口 域名 不同就算是跨域

以下会发生跨域

  • cookie
  • LocalStorage
  • DOM
  • iframe
  • ajax
  • canvas

实现跨域的方法

  • jsonp
  • cors
  • postmessage
  • document.domain
  • window.name
  • location.hash
  • websocket
  • http-proxy
首先是jsonp

jsonp就是利用script标签跨域的特性，服务端发送一段脚本，客户端执行

cors(Cross-Origin Resource Sharing（CORS）跨域资源共享)

cors是一种http标准的跨域手段，同样需要后端设置header来配合

  • res.setHeader(‘Access-Control-Allow-Origin’,orign) //origin是来源地址，最好不要设置为*，否则不能发送凭证
  • res.setHeader(‘Access-Control-Allow-Headers’,’name’) // 允许携带哪个非标准头访问我
  • res.setHeader(‘Access-Control-Allow-Methods’,’put’) // 允许哪些方法访问
  • res.setHeader(‘Access-Control-Allow-Credentials’,true) // 是否允许携带cookie
  • res.setHeader(‘Access-Control-Allow-Max-Age’,’60’) //预检的存活时间，因为设置了允许的方法后，某些请求会发送请求体put，浏览器会发送options方法的http请求测试，如果成功了才会发送请求体过去，这就是设置多少秒发送一个
  • res.setHeader(‘Access-Control-Expose-Headers’,’name’) // 允许浏览器访问的请求头字段

不发送options请求有一些条件

请求方法是以下三种方法之一：

  • HEAD
  • GET
  • POST

HTTP的请求头信息不超出以下几种字段：

  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID ：来声明最后一次接收到的事件的标识符。服务器端可以通过浏览器端发送的事件标识符来确定从哪个事件开始来继续连接。
  • Content-Type：只限于三个值application/x-www-form-urlencoded、multipart/form-data、text/plain

一旦跨域请求不满足条件，就会只发送options请求

通过location.hash跨域

通过window.name进行跨域

window对象有个name属性，该属性有个特征：即在一个窗口(window)的生命周期内,窗口载入的所有的页面都是共享一个window.name的，每个页面对window.name都有读写的权限

configureOrigin

可以通过源码，可以看到, 没有设置origin时Access-Control-Allow-Origin设置为*

还可以传入数组，字符串，正则

origin必须完全匹配，端口也需要匹配

configureCredentials

这个配置比较简单了, 当你跨域发送cookie，设置为true即可

configureMethods

配置允许发送请求的方法

  • 字符串直接返回
  • 数组，join后返回

经过测试GET POST HEAD不设置也没关系

configureAllowedHeaders

配置额外的请求头Access-Control-Request-Headers

通过源码，可以看到，可以通过allowedHeaders与headers进行配置

可以看到，没有设置的话，会把access-control-request-headers设置为返回值

  • 数组，join后返回
  • 字符串直接返回

so, 浏览器在发送options时会发送哪些是额外的请求头

所以，我们设置{headers: []},就不允许请求带上额外的请求头

configureMaxAge

设置预检请求的生存时间

configureExposedHeaders

默认情况下，浏览器只能访问下面几种，通过这个可以配置更多

  • Cache-Control
  • Content-Language
  • Content-Length
  • Content-Type
  • Expires
  • Last-Modified
  • Pragma

例如：正常情况下无法访问Date,可以设置exposedHeaders: 'Date',

  • 数组join后返回
  • 字符串直接返回

非options

可以看到非options请求，只会设置三个请求头

  • origin
  • Credentials
  • ExposedHeaders

综上所述, 需要发送cookie只需要进行下面的设置即可

SameSite策略（跨站请求策略）

首先需要理解csrf攻击原理

首先一个a网站，和一个b网站

a网站里访问b网站，之前默认会携带b网站的cookie

也就是说，你登录了b网站，然后在a网站被构造了恶意请求发往b网站，b网站会认为这是一个合法请求

解决方案很简单，增加一个只有b网站才会有的token, 例如在b网页里嵌入一个token，a网站无法知道这个token

也就无法构造出合法请求

浏览器厂商为了解决这个问题，增加了 SameSite 同源策略

SameSite策略 有三个值可选

  • Strict 最严格，任何情况下访问三方都不会发送cookie
  • Lax 一部分情况可以发送
  • None 关闭改策略

还需要注意，这个是跨站请求策略，不是跨域，也就是 两个网站在同一个顶级域名下，并不会触发该策略
