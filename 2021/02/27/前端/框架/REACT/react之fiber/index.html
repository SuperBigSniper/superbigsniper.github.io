Scheduler

react使用MessageChannel模拟requestIdleCallback

所以Scheduler将需要被执行的回调函数作为MessageChannel的回调执行。如果当前宿主环境不支持MessageChannel，则使用setTimeout

Scheduler内部存在5种优先级

lanePriority

  • 过期任务或者同步任务使用同步优先级
  • 用户交互产生的更新（比如点击事件）使用高优先级
  • 网络请求产生的更新使用一般优先级
  • Suspense使用低优先级
  • React需要设计一套满足如下需要的优先级机制：
  • 可以表示优先级的不同
  • 可能同时存在几个同优先级的更新，所以还得能表示批的概念
  • 方便进行优先级相关计算

lane使用31位二进制来表示优先级车道，共31条, 位数越小（1的位置越靠右）表示优先级越高

计算a、b两个lane是否存在交集，只需要判断a与b按位与的结果是否为0
计算b这个lanes是否是a对应的lanes的子集，只需要判断a与b按位与的结果是否为b
从set对应lanes中移除subset对应lane（或lanes），只需要对subset的lane（或lanes）执行按位非，结果再对set执行按位与。

任务获取赛道的方式是从高优先级的lanes开始的，如果高优先级没有可用的lane了就下降到优先级低的lanes中寻找

怎么解决饥饿问题？ 任务会有过期时间，超过后就会同步执行

hook结构

fiber

react 初始化会创建三个东西

  1. Root 整个应用的根，不是 fiber 是一个对象
  2. current 真的 fiber 树，表示上一个状态 （初始化时没有 current 树，会默认创建一个uninitialFiber树）
  3. workinProcess 表示当前状态的 fiber 树
  4. RootFiber

初始化会创建uninitialFiber，然后让current指向uninitialFiber（他们之间使用 alternate 双向链接）

fiber 节点涨下面这个样子

render阶段：通过render生成新的workinProcess树，然后旧的workProcess就会成为新的current

commit阶段：

  • 执行生命周期
  • 获取 RootFiber 上获取的链表，根据链表的标识

setState 是同步还是异步

  • 正常情况下，是同步更新，但是不会立即获取 state 的值，而是放入 updateQueue 中，事件完成后触发回调函数，更新 state 重新渲染
  • 使用 concurrent 时，同样无法获得最新状态，并且使用了真正的异步方式(postMessage)
  • 当使用了 flushSync，是完全同步的 flushSync(()=>{...})
